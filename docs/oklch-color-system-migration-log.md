# OKLCH Color System Migration Log

Engineering summary by phase. Goals, final decisions, and handoff notes. Process history and intermediate corrections omitted.

---

## Phase 0 — Baseline Audit

**Goal**: Catalog existing HSL-based palette derivations and identify what to harden.

**Outcome**: All palette derivations lived in `SemanticPaletteFactory`. Magic numbers scattered; no token registry; no self-check. Phase 1 targets.

---

## Phase 1 — Token Registry

**Goal**: Extract all magic numbers from `SemanticPaletteFactory` into named constants in `ColorSystemTokens.swift`.

**Landed**:
- `ColorSystemTokens.Accent` — dark/light mode lightness floors and saturation ceilings.
- `ColorSystemTokens.NearMonochrome` — nearMono accent saturation caps.
- `ColorSystemTokens.ReadableText`, `CoverGradient`, `FallbackAccent` — remaining derivation constants.

**Key decision**: Token names reflect semantic intent (`darkMinLByHueCyanBlue`) so the underlying expression can be changed without touching call sites.

---

## Phase 2 — Orthogonal Axes + Structured Palettes

**Goal**: Split `isEffectivelyMonochrome` into two independent gates; add structured palette outputs.

**Landed**:
- `isUltraDark` (lightness axis) — captures "night-feel" covers independently of chromatic confidence. Gate: avgHslL ≤ 0.22 AND wcagLuma ≤ 0.18 AND dominantBrightness ≤ 0.60.
- `isNearMonochrome` (chromatic axis) — four-branch gate on colorfulness/avgSaturation/dominant-bucket. `isEffectivelyMonochrome` is now an alias for backwards compatibility.
- `salientHighlightPalette` — small-area high-visual-impact colors (logo mark, artist highlight). Gated: sat ≥ 0.40, brightness ≥ 0.30, area 1.5–30%, weight ≥ noiseFloor. Scored by weight × (1 + sat × 0.50).
- `displayPalette` — quality-controlled merge of topPalette + salient + rich, capped at 6 (2 on nearMono).

**Key decision — why two separate axes**: A deep violet cover is chromatic (trustworthy hue) but dim. The old combined gate forced a desaturated nearMono accent on it. Separating the axes lets the accent keep its hue while the lightness gate fires independently.

**Key decision — salient priority in displayPalette**: Merge order is top-slot-1 → salient → tail-of-top → rich. Ensures a vivid small-area logo color beats secondary grey tones even under the nearMono `maxCount=2` cap.

## Phase 3 — Consumer Projection (BKArt / Spectrum / Home Shapes)

**Goal**: Wire displayPalette into visual surfaces and add near-mono/ultraDark guards.

**Landed**:
- Spectrum: soft OKLCH chroma shoulder for low-colorfulness inputs; hard clamp to C≤0.010 on nearMono.
- Home Shapes: nearMono → L in [0.05, 0.18] band; ultraDark+nearMono → same band.
- BKArt: ambient background derived from displayPalette dominant with L clamped.

**Key decision — hotfix**: Original consumer projection was missing the nearMono chroma clamp, causing pastel tints on grey covers. Added `isNearMonochrome` branches to both projection helpers and asserted in `ColorSystemSelfCheck`.

---

## Phase 4 — ReadabilityProfile + MiniPlayerControl (OKLCH)

**Goal**: Unify "overlay UI on artwork" foreground decisions; fix nearMono pastel leak on mini-player chrome.

**Landed**:
- `ArtworkReadabilityProfile` — unified foreground for HomeHero overlay, LibraryDetailHeader, and surfaces drawing on top of artwork. Wraps existing HSL derivation in OKLCH nearMono neutraliser (chroma crushed to ≤ 0.004).
- `MiniPlayerControlPalette` — OKLCH-lifted control color for fullscreen mini-player chrome surface. On nearMono: collapses to achromatic warm white at OKLCH(L=0.94, C=0) — fixing the "淡蓝/淡黄" ghost tint on grey covers.
- `SemanticPalette` — both palettes added as first-class fields; consumers read from palette, not from raw analysis.

**Key decision — nearMono control collapse**: The old `resolveControlAccentColor` lifted HSL saturation to ≥ 0.88, amplifying residual nearMono hue into a visible pastel. Fix: bypass the lift on `isNearMonochrome` and emit OKLCH(L=0.94, C=0) directly.

## Phase 4.5 — AppForegroundPalette (Ordinary App UI Text)

**Goal**: Apply a subtly tinted neutral foreground to all ordinary App UI text, threading from a single ThemeStore subscription per high-level view.

### Design

`AppForegroundPalette` — five-tier struct (primary / secondary / tertiary / quaternary / disabled). Generated by `SemanticPaletteFactory.appForeground()`:

- **Hue**: taken from `globalAccent` OKLCH hue.
- **Chroma**: `chromaScale = min(colorfulness / 0.40, 1.0)`, then `c = chromaScale × tierCap × hueChromaFactor`.
- On `isNearMonochrome`: `chromaScale = 0` → all tiers achromatic.
- `disabled`: always C = 0.

### Parameter values (2026-05)

| | Dark primary | Dark secondary | Dark tertiary | Light primary | Light secondary | Light tertiary |
|---|---|---|---|---|---|---|
| L | 0.960 | 0.780 | 0.590 | 0.220 | 0.380 | 0.520 |
| C cap | 0.070 | 0.042 | 0.026 | 0.095 | 0.060 | 0.040 |

**Dark mode hue-aware factor**: blue/cyan (OKLCH hue 0.40–0.72) → 0.65×; violet (0.72–0.88) → 0.75×; warm/other → 1.0×. Blue-white text reads as more "chromatic" than amber-white text at the same OKLCH chroma because the eye adapts to warm neutrals in dark UIs.

**Light mode L correction**: L=0.14 (near-black) made tint imperceptible; the sRGB gamut is nearly zero there. Raised to L=0.22 (dark charcoal) so the chroma is visually detectable while still reading as dark text.

**Secondary/tertiary ratio**: secondary ≈ 60% of primary chroma; tertiary ≈ 62% of secondary. Grey-tier text must not appear chromatic.

### Performance architecture

Parent views subscribe to `ThemeStore` once, pre-resolve `Color(nsColor: themeStore.appForegroundPalette.X)` as `let` bindings, and pass as plain `Color` params to child rows. Row views (`TrackRowView`, `HomeRankRow`, etc.) accept color params with system-color defaults and never subscribe to ThemeStore directly.

### Refresh-delay fix

`TrackRowView.Equatable ==` was missing `rowPrimaryColor` and `rowSecondaryColor`. The `.equatable()` wrapper saw no change when ThemeStore published a new palette, so rows skipped re-render until the user navigated away and back. Fixed by adding all three row color params (`rowPrimaryColor`, `rowSecondaryColor`, `rowTertiaryColor`) to the `==` implementation.

### Self-check coverage (Phase 4.5 assertions)

- Near-mono → all tiers achromatic (C ≤ 0.005)
- Colorful dark primary has detectable tint (C > 0.001, ≤ 0.090)
- Colorful light primary has detectable tint (C > 0.001, ≤ 0.110)
- Light-mode primary C > dark-mode primary C (per-mode cap split)
- Dark-mode L hierarchy: primary > secondary > tertiary > quaternary > disabled
- Light-mode L hierarchy: primary < secondary < tertiary < quaternary < disabled
- AppForeground.primary ≠ ReadabilityProfile.foregroundPrimary (separate pipeline)
- Dark secondary C ≤ primary × 0.65 (ratio cap)
- Dark tertiary C ≤ secondary × 0.70 (ratio cap)
- Cool-hue accent produces lower dark primary C than warm accent (hue-aware reduction)
- Light-mode directional: warm accent R > B; cool accent B > R

## Verification Guide

Test with warm cover (R>B dominant), cool cover (B>R dominant), nearMono cover (R≈G≈B).

1. Home album card title and artist subtitle
2. Home insights stat card label + unit text
3. Home ranking row: song title (primary tint), artist (secondary tint)
4. Sidebar artist/album row text
5. Library track row: title (primary), artist (secondary), duration (tertiary)
6. Settings sidebar: category icon and text
7. Fullscreen queue header title, track title and duration
8. TrackInfoEditor field labels

Expected: warm/cool covers show distinct temperature tilt at each position; nearMono shows neutral grey everywhere; primary most tinted, secondary noticeably less, tertiary very subtle.
