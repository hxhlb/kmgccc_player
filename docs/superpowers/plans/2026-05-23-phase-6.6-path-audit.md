# Phase 6.6 — BKArt / Lyrics Color Path Audit

Audit date: 2026-05-23. Branch: `refactor/oklch-color-system`, HEAD: `f20fbf7`.

## Architecture Overview

```
[PlaybackCoordinator.presentation]
         │
         ├── NowPlayingDisplayContext (FullscreenPlayerView)
         │       ├── BKArtBackgroundView (fullscreen)
         │       │       ├── applySeededPrimaryColorIfNeeded() → BKColorEngine.make(NO analysis)
         │       │       └── applyResolvedPalette() → BKColorEngine.make(WITH analysis)
         │       └── applyFullscreenLyricsTheme() → SemanticPaletteFactory → AMLL
         │
         └── NowPlayingPresentation (AppKitMainSplitPanes)
                 └── BKArtBackgroundView (Now Playing)
                         ├── applySeededPrimaryColorIfNeeded() → BKColorEngine.make(NO analysis)
                         └── applyResolvedPalette() → BKColorEngine.make(WITH analysis)
         
[ThemeStore] (parallel track)
    └── updateThemeFromArtworkData() → analysis → refreshPalette() → SemanticPalette
            └── LyricsSurfaceManager.shared.applyTheme() → window AMLL
```

**Key insight**: BKArt colors and fullscreen lyrics colors resolve their analysis independently from ThemeStore. The FullscreenPlayerView has its own resolution layer with different fallback priorities. ThemeStore is the authority for window AMLL, but fullscreen lyrics can show colors ThemeStore hasn't published yet (or colors that diverge from what ThemeStore published).

---

## Path 1: Paused → Enter Fullscreen BKArt Color Path

1. FullscreenPlayerView appears; `currentDisplayContext` is populated from `playbackCoordinator.presentation.displayContext`
2. Track is paused → `currentDisplayContext.isPlaying = false` (but this only affects NSView animation, not color computation)
3. `BKArtBackgroundView(controller: bkController, trackID:, artworkData: currentDisplayContext.artworkData, isPlaying: false, ..., initialPalette: fullscreenArtBackgroundSeedPalette, holdPaletteWhenArtworkMissing: currentDisplayContext.isArtworkLoading)`
4. **`fullscreenArtBackgroundSeedPalette`**: snapshot.richPalette → snapshot.palette → snapshot.accentColor → snapshot.averageColor → snapshot.dominantColor → **`[themeStore.accentNSColor]`** (line 637)
5. BKArtBackgroundView `.onAppear` → `refreshPalette()`
6. **Step A — `applySeededPrimaryColorIfNeeded()`**: Calls `BKColorEngine.make(extracted: seededPalette, fallback: fallbackPalette, isDark: colorScheme, analysis: nil)` — **NO analysis** → no nearMono neutralization, no `applyAnalysisBgSGating`, `neutraliseShapes = false` unconditionally
7. Controller sets `primaryBackgroundColor` and `isUltraDarkActive` from the seeded harmonized palette
8. **Step B — async extraction**: If ArtworkAssetStore has snapshot → uses snapshot.palette/richPalette/analysis. Otherwise runs `ArtworkColorExtractor.analyze(from: data)` on the background queue
9. **Step C — `applyResolvedPalette()`**: `selectedExtractedPalette(analysis: analysis)` → prefers `analysis.displayPalette` → `richPalette` → `basePalette` → `fallbackPalette`. Then `BKColorEngine.make(extracted: resolvedPalette, fallback:, isDark:, analysis: analysis)` — **WITH analysis** → nearMono neutralization active, `applyAnalysisBgSGating` active
10. Final colors: `stableLyricsBackgroundColor(from: harmonized)` and `primaryBackgroundColor` pushed to controller

**Divergence risk**: Steps A and C produce different colors. Step A has no analysis, so shapes are always colored. Step C with analysis may neutralize shapes for nearMono and apply bgS gating. The user sees Step A colors first, then Step C replaces them.

---

## Path 2: Playing → Enter Fullscreen BKArt Color Path

**Identical to Path 1**, except `currentDisplayContext.isPlaying = true`. The `isPlaying` parameter is passed to `BKArtBackgroundRepresentable` (NSView), not used in the SwiftUI palette computation. The artwork data, extraction, and BKColorEngine paths are the same.

**Conclusion: BKArt colors should NOT differ between paused and playing for the same track.**

---

## Path 3: Paused Now Playing BKArt Color Path

1. AppKitMainSplitPanes creates `BKArtBackgroundView` at line 475
2. `initialPalette: [themeStore.accentNSColor]` — SINGLE accent color (not full palette like fullscreen)
3. No `holdPaletteWhenArtworkMissing` parameter → defaults to `false`
4. `artworkData: playbackCoordinator.presentation.artworkData` — same source as fullscreen
5. Same `refreshPalette()` flow: seeded → async extract → resolved
6. Seeded palette is just `[themeStore.accentNSColor]` — a single hue from whatever ThemeStore has published
7. `BKColorEngine.make(extracted: [accentColor], ...)` — single-color palette input → different `PaletteStats` (single color always produces low-complexity stats) → different `primaryHue` selection → different shapes

**Critical divergence from fullscreen**: The Now Playing seeded palette is a single accent color, while fullscreen seeded palette is the full rich palette. This means the initial (seeded) shapes and background colors are fundamentally different between Now Playing and fullscreen even for the same track.

---

## Path 4: Playing Now Playing BKArt Color Path

**Identical to Path 3** for color computation purposes. The `isPlaying` parameter value has no effect on BKColorEngine.make() or the palette extraction.

---

## Path 5: Pending/Loading BKArt Color Path

There are **three** distinct pending/loading sub-paths:

### 5a. BKArtBackgroundView default state (before any artwork)
- `harmonized` @State: `BKColorEngine.make(extracted: fallbackPalette, fallback: fallbackPalette, isDark: false)` (line 621)
- **Always light mode, always fallback palette (cold blue-gray ~220° hue)**

### 5b. Fullscreen — holdPaletteWhenArtworkMissing = true
- `refreshPalette()` at line 155: "holding previous palette while artwork data is pending"
- Returns without changing palette — previous colors persist

### 5c. Now Playing — holdPaletteWhenArtworkMissing = false (default)
- `refreshPalette()` at line 163: Falls through to `palette = Self.fallbackPalette`
- Resets to cold gray fallback immediately

**This is the "default color flash" during track change!** When `holdPaletteWhenArtworkMissing` is false (Now Playing path), the palette immediately resets to fallback before new artwork data arrives.

## Path 6: Paused Fullscreen Lyrics Color Path

`applyFullscreenLyricsTheme()` orchestrates this. The analysis resolution chain in `resolveLyricsAnalysis()`:

1. If `themeStorePaletteMatchesCurrentArtwork()` → use `themeStore.semanticPalette.analysis`
2. Else if `artworkSnapshot.analysis` exists → use snapshot analysis
3. Else → fallback to `themeStore.semanticPalette.analysis` (which is `.neutralFallback` initially!)

The base color resolution in `resolveFullscreenLyricsBaseColor()`:

1. If `themeStorePaletteMatchesCurrentArtwork()` → use `themeStore.semanticPalette.fullscreenLyricBase`
2. Else if snapshot exists → `snapshot.accentColor ?? snapshot.averageColor ?? snapshot.dominantColor ?? NSColor(AppSettings.shared.accentColor)`
3. Else if `themeStoreArtworkThemePending()` → use `themeStore.semanticPalette.fullscreenLyricBase`
4. **Else → `NSColor(AppSettings.shared.accentColor)`** ← THIS IS THE DANGEROUS FALLBACK

Then `makeFullscreenLyricsColorSet()` passes analysis + colors to `SemanticPaletteFactory.fullscreenLyricsColorSet()` which (when `usesArtisticBackground=true`) calls `artisticFullscreenLyricsColorSet()` → `artisticLyricsSingleSeed()` → `PerceptualToneLadder.artisticLyricsTone()`.

The `artisticLyricsSingleSeed()` seed selection:
1. nearMono without trusted hue → cap chroma, keep hue
2. Salient override via `pickSalientLyricSeed()` (focus score ≥ threshold)
3. Area-dominant if dominantLCH.c ≥ `lyricsDominantSeedMinChroma`
4. Fall through: highest chroma from {topPalette.first, bestTextSourceColor, preferred}

## Path 7: Playing Fullscreen Lyrics Color Path

**Identical to Path 6** for color computation. The `isPlaying` state does not affect the lyrics color resolution chain. The `lockedFullscreenLyricsUltraDark` flag (from `bkController.isUltraDarkActive`) is the only state-dependent variable, and it's gated by `colorScheme == .dark`.

---

## Path 8: Paused/Playing Now Playing Lyrics Color Path

The window AMLL lyrics colors come from `ThemeStore.refreshPalette()` → `LyricsSurfaceManager.shared.applyTheme(newPalette)`. The ThemeStore creates a `SemanticPalette` from its own `analysis` field and uses `semantic.lyrics.windowActive/windowInactive`. This path does NOT go through the fullscreen lyrics resolution — it uses the ThemeStore's own published palette directly.

**Key difference from fullscreen**: The window lyrics always use `ThemeStore.semanticPalette.analysis`, while fullscreen can bypass ThemeStore and use its own snapshot analysis. This means window lyrics and fullscreen lyrics can show different colors for the same track during the gap between snapshot loading and ThemeStore publishing.

---

## Path 9: NearMono Shape Hue — Paused vs Playing

The nearMono shape decision happens in `BKColorEngine.make()`:

```swift
let neutraliseShapes =
    (analysis?.isNearMonochrome ?? false)
    && !analysisHasTrustedHueCandidate(analysis)
```

**When `analysis == nil`** (seeded path, Step A): `neutraliseShapes = false` — shapes are never neutralized, always colored by HSB palette stats.

**When `analysis != nil`** (resolved path, Step C): `neutraliseShapes` is controlled by `analysis.isNearMonochrome` AND whether there's a trusted hue candidate.

The shape hues come from `makeShapePool()` which selects from:
1. `primaryAfter` — the resolved primary hue (from saliencyCandidate → cluster → scorer chain)
2. `hueFamily` — family hues built around the primary
3. `stats.accentHue` / `stats.accentClusterHue` — accent injection
4. `stats.topSalientHues` / `stats.topClusters`

**Paused vs Playing**: No difference. Both go through the same `BKColorEngine.make()` with the same extracted palette. The neutraliseShapes decision depends only on analysis, not on playback state.

**However**, the seeded path (Step A, no analysis) and resolved path (Step C, with analysis) can produce different shapes. If the seeded path runs with a limited palette (e.g., single accent color for Now Playing), the `primaryHue` resolution is different from the full-palette resolved path.

## Root Cause Analysis: Why Same Song Shows Different Colors

### Cause 1: Seeded vs Resolved palette (BKArt)
The seeded path uses `initialPalette` with NO analysis. The resolved path uses the full extracted palette WITH analysis. These produce different `PaletteStats`, different `primaryHue`, different tier ranges, and different shape neutralization.

- **Now Playing**: seeded = `[themeStore.accentNSColor]` (1 color) vs resolved = full palette (4-8 colors) → massive difference
- **Fullscreen**: seeded = full rich palette (similar to resolved) vs resolved = analysis.displayPalette (narrowed) → smaller but real difference

### Cause 2: ThemeStore analysis default flash
`ThemeStore.analysis = .neutralFallback` is the `@Published` initial value. `neutralFallback` has `isNearMonochrome = true`, `isEffectivelyMonochrome = true`, warm amber colors (230/199/153). Any consumer reading `themeStore.semanticPalette.analysis` before the first extraction completes gets this warm amber fallback.

### Cause 3: Fullscreen has independent resolution from ThemeStore
`resolveLyricsAnalysis()` can return snapshot.analysis while ThemeStore still has `.neutralFallback`. `resolveFullscreenLyricsBaseColor()` can return snapshot colors while ThemeStore still has old palette. The fullscreen and window AMLL can show different colors for the same track.

### Cause 4: Fallback chain terminates at AppSettings.accentColor
`resolveFullscreenLyricsBaseColor()` line 3636: final fallback is `NSColor(AppSettings.shared.accentColor)` — this is a user-chosen accent color, completely unrelated to artwork. When neither ThemeStore nor snapshot can provide a color, fullscreen lyrics use the app accent color as a seed, producing colors unrelated to the artwork.

### Cause 5: Now Playing holdPaletteWhenArtworkMissing defaults to false
AppKitMainSplitPanes doesn't pass `holdPaletteWhenArtworkMissing:`, so it defaults to `false`. When artwork data is nil during track change, the palette immediately resets to cold gray fallback. Fullscreen passes `holdPaletteWhenArtworkMissing: currentDisplayContext.isArtworkLoading` which may be `true` during loading.

### Cause 6: Default harmonized @State is always light mode
`@State private var harmonized: HarmonizedPalette = BKColorEngine.make(extracted: fallbackPalette, fallback: fallbackPalette, isDark: false)` — line 621. Even in dark mode, the initial BKArt background is light-mode cold blue. This shows as a flash before the first `refreshPalette()` call.

## Summary: Paths That Actually Diverge

| Aspect | paused vs playing | Same? |
|--------|-------------------|-------|
| BKArt colors (any view) | Uses same artwork data, same BKColorEngine path | YES |
| Fullscreen lyrics colors | Uses same resolution chain | YES |
| Now Playing lyrics colors | Uses same ThemeStore palette | YES |
| NearMono shapes | Uses same analysis, same BKColorEngine | YES |

**paused vs playing is NOT the divergence.** The actual divergences are:

| Aspect | Divergence |
|--------|-----------|
| Seeded vs resolved BKArt | Different palettes, different analysis (nil vs real) |
| Now Playing vs Fullscreen BKArt seeded | 1 color vs full palette → different stats → different shapes |
| ThemeStore vs Fullscreen analysis | ThemeStore may have neutralFallback while fullscreen has real analysis |
| Track change default flash | holdPaletteWhenArtworkMissing=false resets to fallback |
| Default harmonized state | Always light mode fallback, even in dark mode |

## Resolution — Fixes Applied（2026-05-23）

Based on this audit, the following fixes were implemented and verified (build + 61/61 self-check PASS):

1. **Track-change default color flash** — Coordinated fix across three sites:
   - `AppKitMainSplitPanes.swift` line 484: added `holdPaletteWhenArtworkMissing: playbackCoordinator.presentation.isArtworkLoading` to Now Playing `BKArtBackgroundView`.
   - `ThemeStore.swift`: `updateThemeFromArtworkData()` holds current theme via `holdCurrentThemeForPendingArtwork()` when artwork data is nil and source changed, instead of resetting to `.neutralFallback`.
   - `PlaybackCoordinator.swift`: `isArtworkLoading` derived from `track.artworkData?.isEmpty != false && track.resolvedArtworkURL() != nil` (was hardcoded `false`).

2. **Paused/playing path unification** — Audit confirmed no code changes needed. The `isPlaying` parameter does not affect color computation in any path. Marked complete.

3. **nearMono shapes pink fix** — `BKColorEngine.neutraliseCGColor()` rotates warm residual hues to `ColorSystemTokens.NearMonochrome.neutralCoolHue` (~209° cyan-blue) for true nearMono shapes, replacing uncontrolled residual pink.

4. **Daytime brightness fine-tune** — `BKColorEngine.tierRanges()` `lowSatColorCover` block: `fgB` and `dotB` brightness boost gated to `isDark` only. Prevents light-mode shapes/background from being pushed to pure white (1.0) and becoming invisible.

5. **Documentation updated** — `docs/oklch-color-system-migration-log.md` and `docs/oklch-color-system-execution-plan.md` amended with Phase 6.6 entry.
